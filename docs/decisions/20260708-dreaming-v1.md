# Dreaming v1 — nightly knowledge consolidation (the `dreaming` module)

**Date:** 2026-07-08 · **Status:** shipped (feat/dreaming-v1) ·
**Plan:** `docs/plans/active/20260707-dreaming-and-knowledge-tree.md` (internal)

## Context

The last unshipped Show-HN claim ("nightly dreaming + a Knowledge tab") and,
simultaneously, rung 2 of the AI-org staircase: *folder = project = AI
employee* only works if an agent spawned in a folder arrives knowing that
project. Four production memory systems (two in-house stacks and two open-source
assistants) were studied end-to-end first; their converged lessons are the
design.

## The decision

One **host-orchestrated** nightly sweep (`api/services/dreaming.py`, systemd
`shellteam-dream.timer` at 03:30, `Persistent=true`), never one scheduler per
folder, and the LLM never orchestrates — it only proposes:

1. **Gather** new sessions since a watermark from each agent's own transcripts
   (Claude Code JSONL, Codex rollouts, OpenCode SQLite), tool noise stripped.
2. **Route by cwd** to a node of the shadow knowledge tree
   (`~/.shellteam/knowledge/tree/<relpath>/`), which mirrors the home folders.
3. **Extract** per node-batch: one headless `claude -p` call **on the owner's
   subscription** (`ANTHROPIC_API_KEY` stripped from the subprocess env — the
   prefer-subscription rule) proposing a strict JSON delta.
4. **Apply deterministically** (`api/services/knowledge_tree.py`):
   propose-then-apply with structural scope enforcement (a node batch cannot
   write another node; deletes don't exist as an op), dedup, index size cap
   with overflow to `details/archive.md`, corrections appended never
   overwritten, `changelog.jsonl`, high-confidence auto / rest → review queue.
5. **Report**: HTML dream report in `~/reports/`, per-run audit artifacts
   under `~/.shellteam/dream/runs/<stamp>/`, and a morning ping through the M0
   notify primitive. Failures are loud (non-zero oneshot + failure report);
   the watermark advances **only on success**.

**Spawn-time composition** (the AI-employee mechanic): with the module on,
`agent-layer.mjs` composes user layer + the cwd's node chain (ancestors, never
siblings; cross-project map only at root scope) into the runtime layer —
Claude via `--append-system-prompt-file`, OpenCode via `instructions`. Pure
core and persona-only boxes spawn byte-identically (contract-tested).

## Deliberate v1 choices (and their revisit triggers)

- **Hygiene is report-only** (stale tabs, old ~/tmp surfaced, nothing
  deleted). ShellTeam deleting user data autonomously needs an explicit opt-in
  knob, not a default. Revisit: add `DREAM_PRUNE_*` knobs when asked.
- **Codex gets no knowledge injection yet** — it has no verified
  instructions-file mechanism in this codebase (`-c` overrides only; the
  doc-fallback reads the project's own AGENTS.md, which we will not write to).
  Revisit: test `experimental_instructions_file` upstream.
- **Antigravity transcripts are not ingested** — proprietary binary trajectory
  format, and its summary index is empty on real boxes. Revisit when `agy`
  ships a readable export.
- **Single watermark, no per-file cursors; no promotion ladder** — the most mature
  studied system's robustness is earned complexity; v1 takes the cloud-style simple cutoff
  plus the review queue. Revisit at the first real dedup/staleness incident.
- **Extraction engine is the claude CLI only** — no silent fallback chain. If
  it fails, the run fails loudly. Revisit: engine choice in Settings.
- **Default model is Opus, and `--full` includes dreaming** (both changed
  2026-07-08 on Seb's call). Because extraction rides the owner's Claude
  subscription and runs once nightly, the sweep is the one place worth the
  best model. Dreaming is core to the harness ("every folder an employee who
  remembers"), so the full experience must include it — not an à-la-carte
  extra. Revisit only if nightly Opus spend becomes a problem on a metered
  plan (then: expose `DREAM_MODEL` in Settings / gate `--full` inclusion).
- **User-layer split** — identity/preferences/feedback inject everywhere;
  projects/contacts only at root scope. If a folder ever maps to an external
  client (AI-org M2+), per-layer forbidden lists already make wrong-layer
  writes structurally impossible.

## Consequences

- The Show-HN post's dreaming claim is now real, module-gated, and
  footprint-documented (FOOTPRINT.md, uninstall.sh teardown).
- The AI-org plan's "knowledge & memory hangs off the folder key" is
  implemented; employee cockpits (M0) can get project knowledge by building
  the layer with `dreaming` enabled once a container-safe knowledge scope is
  decided (M1 work).
- New owner surface: dashboard **Knowledge tab** (read/edit every file, review
  queue, dream-now) — the "what does my computer know about me" transparency
  story.
